# Plan: Split Frontend and Backend onto Different Servers

## Objective
Split the Driver Tracker application into frontend (Next.js) and backend (NestJS/Express API) services, deploying them on separate servers for better scalability, security, and maintainability.

## Development Branch Strategy

All work for this split will happen on a long‑lived `dev` branch, not on `main`:

1. Create the `dev` branch from `main`:
   ```bash
   git checkout main
   git pull origin main
   git checkout -b dev
   git push -u origin dev
   ```
2. For each sub‑task (backend extraction, frontend changes, Docker changes, etc.) create a short‑lived feature branch off `dev`:
   ```bash
   git checkout dev
   git checkout -b feat/backend-scaffold
   # ...work, commit...
   git push -u origin feat/backend-scaffold
   # open PR into dev
   ```
3. Only merge `dev` into `main` once the full split is verified end‑to‑end (lint, build, manual QA, customer smoke test).
4. Keep `main` deployable at all times; use `dev` for the Coolify staging application (separate Coolify app pointing to branch `dev`).
5. Tag a release (`v-pre-split`) on `main` before the first merge back so rollback is trivial.

**Coolify implications:**
- Staging Coolify app (frontend + backend) tracks branch `dev` with auto‑deploy.
- Production Coolify app continues to track `main` until cutover.
- Database for `dev` should be a separate PostgreSQL resource (never point `dev` at the production DB).

## Current Architecture Analysis
- Monolithic Next.js application with API routes in `src/app/api/`
- Prisma ORM for database access
- NextAuth for authentication
- Dockerized deployment

## Proposed Architecture
### Frontend Server
- Next.js application (pages, components, styles)
- Static asset serving

### Backend Server

## Implementation Steps

4. Extract Prisma client initialization to backend
5. Move authentication logic to backend

### 2. Frontend Modifications
1. Create `frontend/` directory (or keep current structure)
2. Replace direct API calls with HTTP client (Axios/Fetch) to backend URL
3. Update environment variables to point to backend API (NEXT_PUBLIC_API_URL, NEXTAUTH_URL, NEXTAUTH_SECRET)
4. Remove API route files (`src/app/api/*`) as they'll be handled by backend
5. Keep pages, components, and styling intact
6. Create Dockerfile for frontend service

### 2a. Auth Token Handling (NextAuth → Backend JWT)
- Options:
  1. Cookie-based session (recommended for SSR): backend sets secure HttpOnly cookie(s) on login. Frontend sends cookies automatically; protects against XSS. Ensure CORS credentials allowed when frontend/backend are on different origins.
  2. Bearer token (localStorage/sessionStorage): backend issues JWT; frontend stores in memory or secure storage and adds `Authorization: Bearer <token>` to requests. Simpler for SPA ↔ API on different subdomains but requires XSS mitigation and CSRF protection if cookies also used.
- Migration steps:
  - Keep NEXTAUTH_SECRET on backend for token signing.
  - Implement /api/auth/* endpoints on backend (sign-in, sign-out, csrf, session refresh).
  - Frontend replaces next-auth client usage with custom hook/service that reads token from cookie or storage and attaches to requests.
  - Set NEXTAUTH_URL on backend to frontend origin for OAuth callbacks if used.

### 2b. File Upload / Download Strategy
- Uploads: frontend sends multipart/form-data to backend endpoint (e.g., POST /api/checkins/:id/attachments). Backend validates, stores to disk/S3-compatible storage, records metadata in DB, returns URL.
- Downloads: backend serves signed URLs or streams file with proper auth checks. For large files consider redirect to signed S3 URL.
- If storing locally, ensure persistent volume on backend server; if using S3/compatible, use presigned URLs to avoid proxying large files through backend.

### 2c. CORS and Security Headers
- Backend must allow frontend origin(s) only (dev/staging/prod) and set Access-Control-Allow-Credentials: true if using cookies.
- Apply common security headers (HSTS, X-Content-Type-Options, Referrer-Policy) at the frontend server or reverse proxy (Nginx) layer.

### 2d. Caching Strategy & Latency Mitigation
- API-level caching (Redis) for GET endpoints that are idempotent (driver/location lists).
- Short-lived client-side cache (SWR/React Query) to reduce roundtrips for frequently accessed data.
- Compress responses (gzip/br) and enable HTTP/2 on backend.
- Keep critical flows (check‑in/out) real‑time with minimal caching; use WebSockets or server‑sent events for urgent notifications if needed.

### 2e. Frontend Deployment Alternatives

A. Docker (local testing)
  - Single-container Next.js build (next build && next start).
  - Expose port 3000; set NEXT_PUBLIC_API_URL=http://localhost:3001 (or backend container name in compose) so API calls go to backend.
  - Works well with docker‑compose for quick dev/staging runs.

B. Standalone static/SSR server with Nginx (on‑server)
  - Build the Next.js app (next build && next export for fully static, or keep SSR with next start).
  - Nginx serves static assets and proxies /api/* to backend (or frontend can call backend directly by host/port).
  - Recommended Nginx minimal config (SSR proxy example):
    server {
      listen 80;
      server_name tracker.example.com;
      location / { proxy_pass http://localhost:3000; proxy_set_header Host $host; }
      location /api/ { proxy_pass http://localhost:3001; proxy_set_header Host $host; }
    }
  - Use systemd to keep the Node process running if using SSR without Docker.

C. Static hosting / CDN (future option)
  - Export or build for Vercel/Netlify/CDN; backend runs as separate origin and frontend uses NEXT_PUBLIC_API_URL=https://api.example.com.
  - Requires proper CORS on the backend and cookie/JWT handling (see Auth section).

Start with A for initial testing, migrate to B for the customer test server, and consider C for eventual production frontend hosting.

### 3. Database Configuration
- Single PostgreSQL instance accessible by both servers
- Connection string managed via environment variables
- Consider read replicas for scaling if needed

### 4. Deployment Strategy

#### Option A: Docker Compose (Development/Local Testing)
- Place frontend, backend, and PostgreSQL in one docker-compose.yml.
- Networks: isolated `frontend` and `backend` networks; shared `db` service.
- Frontend container: port 3000 mapped to host. Backend container: port 3001 (internal only).
- Quick setup for dev and for local verification before pushing to `dev` branch.
- Example services:
  - frontend: build: ./frontend (or . if keeping monorepo structure), depends_on: [backend]
  - backend: build: ./backend, expose: "3001", environment: DATABASE_URL=postgres://...
  - db: image: postgres:15-alpine, volumes: pgdata:/var/lib/postgresql/data

#### Option B: Independent Deployments (Coolify Staging — dev branch)
- Use a **staging Coolify application for frontend** (tracks branch `dev`) and a **separate Coolify resource for backend API** (also tracking `dev`).
- Alternatively, deploy both frontend and backend as separate services under one Coolify app but with distinct internal ports/routing (easier to share network internally).
- Use a separate PostgreSQL resource (`dev-postgres`) created in Coolify; never point `dev` at production DB.
- Frontend env: NEXT_PUBLIC_API_URL=https://backend-staging.example.com (or internal service name if both in same Coolify private network), NEXTAUTH_URL=https://frontend-staging.example.com.
- Backend env: DATABASE_URL from Coolify-managed PostgreSQL, CORS_ORIGIN=https://frontend-staging.example.com, NODE_ENV=production.
- Enable auto-deploy on push to `dev` for both frontend and backend; use Coolify health checks; set up staging notifications (Discord/email).

#### Option C: Independent Deployments (Production)
- Frontend: deployed to CDN/static host with SSR or as a containerized service behind Nginx/LoadBalancer.
- Backend: deployed to dedicated VPS/container platform (Coolify, ECS, etc.) with proper scaling, backups, and monitoring.
- Database: managed PostgreSQL with backups, read replicas (if needed), and restricted access (backend IP only).
- Secrets managed via environment variables or secret manager; rotate keys before cutover.
- Traffic routing: DNS points frontend to CDN/Load balancer; backend served from api.example.com with TLS and WAF.
- Monitoring: centralized logs (ELK/Datadog), APM, alerts for latency/errors; synthetic monitoring for critical user flows.

### 6. Security Considerations
- Restrict backend access: allow only frontend origins/IPs (CORS) and, if possible, use VPC/private networking between frontend and backend in staging/production.
- HTTPS everywhere (TLS termination at CDN/Load balancer or Nginx).
- Rate limiting and authentication on backend (express-rate-limit / NestJS rate limiter).
- Secure cookies if using cookie-based sessions (HttpOnly, Secure, SameSite=Strict/Lax).
- Regular secret rotation and vulnerability scanning of dependencies/images.
- Database connection encryption (SSL) and least-privilege DB roles.

## Risks and Mitigations
1. **Data inconsistency during migration** - Use database transactions and backups
2. **Breaking API changes** - Version APIs and maintain backward compatibility
3. **Increased latency** - Optimize API calls, consider caching
4. **Operational complexity** - Invest in monitoring and logging infrastructure
5. **Security misconfiguration** - Regular security audits and penetration testing
