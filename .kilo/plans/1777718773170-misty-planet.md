# Coolify Backend Testing Setup Plan

## Project Context
- **Original**: Monolithic Next.js + Express app (in `_archive_pre_split/`) with all auth, checkin, admin, SMS, cron, and notification features
- **Current**: Split into Frontend (Next.js PWA) + Backend (Express API) - backend is INCOMPLETE
- **Goal**: Complete the backend to match original functionality, then deploy to Coolify for testing
- **User preference**: No sign-up (users created from backend only), frontend handled separately

## Current State Analysis

### What's in the Original Monolithic App (`_archive_pre_split/`)
Complete API with all features:
- **Auth**: NextAuth with credentials provider, JWT tokens, bcrypt password hashing
- **Checkin**: Create, read, checkout, idempotency keys, extended stay, geocoding
- **Checkins**: Admin list with pagination, driver filtering, date ranges
- **Locations**: Full CRUD with admin-only access
- **Users**: Admin CRUD (list, create, update, delete) with role management
- **Admin Stats**: Dashboard counts (drivers, locations, checkins, active)
- **SMS**: Twilio integration for manual triggers and inbound webhooks
- **SMS Audit**: Full audit trail with inbound/outbound logging
- **Settings**: Notification settings (Twilio, Email) with encrypted storage
- **Cron**: Background job every 15 min for alert escalation (3 tiers, extended stay support)
- **Notifications**: Email (Microsoft Graph) and SMS (Twilio) alert helpers
- **Offline**: Service worker, IndexedDB queue, idempotency for checkin/checkout

### What's in the Split Backend (`backend/`) Currently
Incomplete - only has:
- **Locations**: CRUD routes (GET, POST, PUT, DELETE) - working
- **Auth**: JWT middleware, RBAC helpers (no login route)
- **Prisma**: Client setup, schema with all models
- **Server**: Express on port 3001
- **Dockerfile**: Multi-stage build
- **Skeleton**: `src/checkin/checkin.controller.ts`, `src/checkin/checkin.service.ts` - NOT connected

### What's MISSING in Split Backend
1. **Auth Routes** - Login endpoint (POST /api/auth/login), no signup (correct per user request)
2. **Checkin Routes** - POST /api/checkin (create), GET /api/checkin (current), POST /api/checkout
3. **Checkins Routes** - GET /api/checkins (admin list with filters)
4. **Users Routes** - GET/POST /api/admin/users, PUT/DELETE /api/admin/users/[id]
5. **Stats Routes** - GET /api/admin/stats
6. **SMS Routes** - POST /api/admin/sms/trigger, GET /api/admin/sms-audit
7. **Settings Routes** - GET/PUT /api/admin/settings/notifications
8. **Webhook Routes** - POST /api/webhooks/twilio/inbound-sms, POST /api/webhooks/twilio/sms-status
9. **Cron Jobs** - Background alert escalation (node-cron)
10. **Helper Libraries** - settings-crypto, notifications, notification-settings, sms-failsafe
11. **Database Seed** - prisma/seed.ts for initial admin user and test data

## Implementation Plan

### Phase 1: Complete Backend Implementation (All Missing Features)

#### 1.1 Create Helper Libraries
Copy from original to backend:
- `backend/lib/settings-crypto.ts` - AES-256-GCM encryption for secrets ✓ from original
- `backend/lib/notification-settings.ts` - CRUD for notification settings with encryption ✓ from original  
- `backend/lib/notifications.ts` - Email (MS Graph) and SMS (Twilio) senders ✓ from original
- `backend/lib/cron.ts` - Alert escalation job (15 min intervals, 3 tiers, extended 3x) ✓ from original
- `backend/lib/sms-failsafe.ts` - Config constants for SMS limits ✓ from original
- `backend/lib/offline/` - IndexedDB queue for offline actions (frontend only, skip for backend)

#### 1.2 Add Database Seed
- `backend/prisma/seed.ts` - Create default admin, test driver, test locations
- Update package.json: add "seed": "ts-node prisma/seed.ts" script
- Update Dockerfile to run seed on startup if DB empty

#### 1.3 Implement Auth System
- Create `backend/routes/authRoutes.ts`:
  - POST /api/auth/login - Validate credentials with bcrypt, return JWT (NEXTAUTH_SECRET)
  - POST /api/auth/me - Get current user (protected)
  - No signup endpoint (users created by admin only, per user request)
- Integrate with existing JWT middleware in `server.ts`
- Use bcryptjs for password hashing (already in package.json)

#### 1.4 Implement Checkin Routes
- Create `backend/routes/checkinRoutes.ts`:
  - POST /api/checkin - Create checkin with idempotency, extended stay, location validation
  - GET /api/checkin - Get current driver's open checkin
  - POST /api/checkout - Close checkin with idempotency
- Connect to `src/checkin/checkin.controller.ts` or create new controllers
- Use Prisma with includes for location data

#### 1.5 Implement Checkins Routes (Admin)
- Create `backend/routes/checkinsRoutes.ts`:
  - GET /api/checkins - Admin list with pagination, filters (driver, date range, status, location)
- Use existing `src/checkin/checkin.service.ts` or create new

#### 1.6 Implement Users Routes (Admin)
- Create `backend/routes/usersRoutes.ts`:
  - GET /api/admin/users - List all (superuser sees all, admin sees non-superuser)
  - POST /api/admin/users - Create user with bcrypt password hashing
  - PUT /api/admin/users/[id] - Update user
  - DELETE /api/admin/users/[id] - Delete (cannot delete self or superuser unless superuser)
- Use bcryptjs for password hashing

#### 1.7 Implement Stats Routes (Admin)
- Create `backend/routes/statsRoutes.ts`:
  - GET /api/admin/stats - Return counts: totalDrivers, totalLocations, activeCheckIns, totalCheckIns

#### 1.8 Implement SMS Routes (Admin)
- Create `backend/routes/smsRoutes.ts`:
  - POST /api/admin/sms/trigger - Manual SMS send via Twilio
  - GET /api/admin/sms-audit - Send SMS via Twilio (or fetch audit log)
- Use Twilio SDK (add `twilio` to package.json)

#### 1.9 Implement Settings Routes (Superuser)
- Create `backend/routes/settingsRoutes.ts`:
  - GET /api/admin/settings/notifications - Get masked settings
  - PUT /api/admin/settings/notifications - Update settings with encryption
- Integrate notification-settings helper

#### 1.10 Implement Webhook Routes
- Create `backend/routes/webhookRoutes.ts`:
  - POST /api/webhooks/twilio/inbound-sms - Parse SMS commands (CHECKIN, CHECKOUT), create checkins
  - POST /api/webhooks/twilio/sms-status - Handle delivery status
- Add MAX_SMS_ACTIONS_PER_DAY limit per driver

#### 1.11 Update Server Entry Point
- Update `backend/server.ts`:
  - Import and use all new routes
  - Start cron job on startup (if not in test mode)
  - Add error handling middleware
