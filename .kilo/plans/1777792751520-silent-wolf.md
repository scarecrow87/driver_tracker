# Plan: Create docker-compose.dev.yaml with Nginx for Local Frontend Testing

## Objective
Create a `docker-compose.dev.yaml` file at the project root to run the full stack locally with the frontend served via nginx, enabling production-like local testing of the Next.js PWA.

## Background & Analysis

### Current State
- The project is split into two services: `frontend/` (Next.js PWA) and `backend/` (Node.js/Express API)
- Existing `docker-compose.yml` only includes `db` (Postgres) and `backend` services
- Frontend uses Next.js 14 with `output: 'standalone'` in `frontend/next.config.js`
- Frontend dependencies: `next`, `next-auth`, `next-pwa`, React, Leaflet
- Frontend environment variables:
  - `NEXTAUTH_URL` — used by NextAuth (runtime)
  - `NEXTAUTH_SECRET` — session encryption (runtime)
  - `NEXT_PUBLIC_API_URL` — base API URL for client-side fetch calls (build-time, exposed to browser)
- Frontend local dev currently runs via `npm run dev` (no nginx)

### Production Pattern
The deployment documentation (`frontend/docs/nginx-frontend-config.md`) describes production deployment where:
- Nginx runs as reverse proxy on port 80
- Static assets served directly from `.next/static/` and `public/`
- All other requests proxied to Node.js server (`server.js`) running on localhost:3000
- API calls can optionally be proxied through nginx to backend (`/api/` → `http://backend:3001/api/`)

The plan is to replicate this production pattern in a local Docker Compose setup.

## Decision

**API Routing**: Separate origins. Frontend and backend run on different hostnames/ports.

- Local development: Frontend at `http://localhost:3000`, backend at `http://localhost:3001/api`. Frontend calls backend directly (no nginx API proxy).
- Production: Frontend domain (e.g., `https://pelz-dt.shellshock.net.au`) calls backend domain (e.g., `https://api.pelz-dt.shellshock.net.au/api`). Backend must enable CORS for frontend origin.

## Proposed Implementation

### File 1: `frontend/Dockerfile` (new)
A multi-stage Dockerfile:

**Stage 1 – Builder**
- Base image: `node:20-alpine`
- Working directory: `/app`
- Copy `frontend/package.json` and `package-lock.json` (or just `package*.json`)
- Run `npm ci` (clean install)
- Copy entire `frontend/` directory
- Accept `NEXT_PUBLIC_API_URL` as a build argument (baked into JS bundle)
- Run `npm run build` — produces `.next/standalone/` and `.next/static/`

**Stage 2 – Runtime**
- Base image: `node:20-alpine` (includes Node.js to run the standalone server)
- Working directory: `/app`
- Copy standalone output from builder: `.next/standalone/` → `/app/`
- Copy static assets: `.next/static/` → `/.next/static/`
- Copy `public/` → `/app/public/`
- Install nginx: `apk add --no-cache nginx`
- Copy custom nginx config: `frontend/nginx.conf` → `/etc/nginx/nginx.conf`
- Copy a custom entrypoint script `frontend/docker-entrypoint.sh` → `/usr/local/bin/` and make it executable
- Expose port `80` (nginx) and `3000` (Node, for nginx upstream; not published externally)
- Entrypoint: `docker-entrypoint.sh`

**Rationale**
- Using `node:20-alpine` keeps image small
- Nginx is installed in the final image; it runs alongside Node (both in same container)
- Node runs on port 3000 internally; nginx listens on port 80 and proxies dynamic requests to Node
- This matches the production architecture where nginx and Node run on the same host

**Important**: `NEXT_PUBLIC_API_URL` is needed at build time because it is inlined into client bundles. We'll pass it as a build argument in docker-compose.

### File 2: `frontend/nginx.conf` (new)
Standard nginx configuration for Next.js standalone:

```nginx
worker_processes 1;
error_log /var/log/nginx/error.log warn;
pid /var/run/nginx.pid;

events {
    worker_connections 1024;
}

http {
    include /etc/nginx/mime.types;
    default_type application/octet-stream;
    sendfile on;
    keepalive_timeout 65;

    # Compression
    gzip on;
    gzip_vary on;
    gzip_min_length 1024;
    gzip_proxied any;
    gzip_types text/plain text/css text/xml text/javascript application/javascript application/json application/xml;

    server {
        listen 80;
        server_name localhost;
        root /app;
        index index.html;

        # Security headers
        add_header X-Frame-Options "SAMEORIGIN" always;
        add_header X-Content-Type-Options "nosniff" always;
        add_header X-XSS-Protection "1; mode=block" always;

        # Cache static assets (images, fonts, etc.)
        location ~* \.(js|css|png|jpg|jpeg|gif|ico|svg|woff|woff2|ttf|eot)$ {
            expires 1y;
            add_header Cache-Control "public, immutable";
            try_files $uri =404;
        }

        # Next.js static build files
        location /_next/static/ {
            alias /app/.next/static/;
            expires 1y;
            add_header Cache-Control "public, immutable";
            try_files $uri =404;
        }

        # PWA icons
        location /icons/ {
            alias /app/public/icons/;
            expires 1y;
            add_header Cache-Control "public, immutable";
        }

        # Manifest
        location /manifest.json {
            alias /app/public/manifest.json;
        }


        # Main application — proxy to Next.js Node server
        location / {
            proxy_pass http://127.0.0.1:3000;
            proxy_http_version 1.1;
            proxy_set_header Host $host;
            proxy_set_header X-Real-IP $remote_addr;
            proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
            proxy_set_header X-Forwarded-Proto $scheme;
            proxy_set_header Upgrade $http_upgrade;
            proxy_set_header Connection "upgrade";
        }
    }
}
```

**Notes**
- `root /app;` points to the working directory where `server.js` lives (standalone output root)
- The alias paths carefully map to the actual file locations in the container
- The `location /` block proxies all non-static requests to the Node server (port 3000) to handle SSR and routes; API calls are made directly from the client to the backend using `NEXT_PUBLIC_API_URL`.

### File 3: `frontend/docker-entrypoint.sh` (new)
Simple shell script to startup both nginx and Node:

```sh
#!/bin/sh
# Start nginx in background (foreground mode)
nginx -c /etc/nginx/nginx.conf -g 'daemon off;' &

# Start the Next.js server
exec node server.js
```

- The `exec` ensures Node becomes PID 1 so signals are handled properly
- Nginx runs as a background process; when Node exits, the container exits

### File 4: `docker-compose.dev.yaml` (new at project root)
Compose file defining the development stack:

```yaml
version: '3.8'

services:
  db:
    image: postgres:15-alpine
    container_name: driver_tracker_db
    restart: unless-stopped
    environment:
      POSTGRES_USER: ${POSTGRES_USER:-postgres}
      POSTGRES_PASSWORD: ${POSTGRES_PASSWORD:-password}
      POSTGRES_DB: ${POSTGRES_DB:-driver_tracker}
    volumes:
      - postgres_data:/var/lib/postgresql/data
    ports:
      - "5432:5432"
    healthcheck:
      test: ["CMD-SHELL", "pg_isready -U postgres"]
      interval: 10s
      timeout: 5s
      retries: 5

  backend:
    build:
      context: ./backend
      dockerfile: Dockerfile
    container_name: driver_tracker_backend
    restart: unless-stopped
    ports:
      - "3001:3001"
    environment:
      DATABASE_URL: postgresql://${POSTGRES_USER:-postgres}:${POSTGRES_PASSWORD:-password}@db:5432/${POSTGRES_DB:-driver_tracker}
      NODE_ENV: ${NODE_ENV:-production}
      # Add other backend env vars as needed (Twilio, etc.)
    depends_on:
      db:
        condition: service_healthy

  frontend:
    build:
      context: ./frontend
      dockerfile: Dockerfile
      args:
        # Build-time arg: embedded into the JS bundle
        NEXT_PUBLIC_API_URL: ${NEXT_PUBLIC_API_URL:-http://localhost:3001/api}
    container_name: driver_tracker_frontend
    restart: unless-stopped
    ports:
      - "3000:80"   # Host 3000 → Container nginx 80
    environment:
      # Runtime environment variables
      NEXTAUTH_URL: http://localhost:3000
      NEXTAUTH_SECRET: ${NEXTAUTH_SECRET:-dev-secret-change-me}
      # NEXT_PUBLIC_API_URL is already baked; no need to set at runtime unless server-side code uses it
    depends_on:
      - backend
      - db

volumes:
  postgres_data:
```

**Explanation**
- The `db` service is identical to the existing `docker-compose.yml`
- The `backend` service builds from `backend/Dockerfile`, exposing port 3001
- The `frontend` service builds from new `frontend/Dockerfile`, mapping host port 3000 to nginx port 80 inside container
- Build args pass `NEXT_PUBLIC_API_URL` into the build; default is `http://localhost:3001/api` (direct connection from browser to backend)
- Runtime env provides `NEXTAUTH_URL` and `NEXTAUTH_SECRET` (the latter should be overridden with real secret)
- The frontend depends on backend and db (start order only)

### Optional Enhancements

1. **Healthchecks**
   - Add a healthcheck to `backend` (if not already present) to ensure it's ready before frontend starts
   - Could add a simple `curl` to `/health` endpoint if available

2. **API Proxy through Nginx**
   - If you want same-origin API calls (no CORS, matches production), enable the `/api/` block in `frontend/nginx.conf` and set `NEXT_PUBLIC_API_URL=/api` as build-arg
   - Then client fetches to `/api/...` go to nginx → backend

3. **Hot-reload Development Mode**
   - This setup is for production-like local testing (built assets)
   - For active development with hot reload, use `npm run dev` separately; this compose file is for end-to-end integration testing

## Verification Steps

1. **Build and start**
   ```bash
   docker compose -f docker-compose.dev.yaml up --build
   ```

2. **Access the application**
   - Frontend: http://localhost:3000
   - Backend API: http://localhost:3001/api

3. **Test core flows**
   - Login as superuser (use seeded credentials or create via backend)
   - Navigate to driver dashboard, check in/out
   - Navigate to admin dashboard, view stats, manage locations/users
   - Verify PWA install banner appears (if applicable)

4. **Validate nginx is serving**
   - Check response headers for static assets include `Cache-Control: public, immutable`
   - Check security headers present (X-Frame-Options, etc.)
   - Disable Node process in container to confirm nginx returns 502 for dynamic routes (optional)

5. **Check logs**
   - `docker logs driver_tracker_frontend` should show both nginx and node startup
   - Ensure no errors about missing files (`.next/static`, `public`)

6. **Stop and clean**
   ```bash
   docker compose -f docker-compose.dev.yaml down
   # To preserve DB volume, omit -v; to wipe DB: docker compose -f docker-compose.dev.yaml down -v
   ```

## Potential Pitfalls & Mitigations

| Issue | Cause | Mitigation |
|-------|-------|------------|
| Frontend shows blank page | Node server not started or crashed | Check container logs; ensure entrypoint script runs both processes |
| Static assets 404 | Wrong path in nginx `alias` directives | Verify paths: `/app/.next/static/` and `/app/public/` exist |
| API calls fail (CORS) | `NEXT_PUBLIC_API_URL` points to wrong host/port | Use `http://localhost:3001/api` for direct backend access; ensure backend CORS allows `http://localhost:3000` or use nginx proxy |
| Auth redirects loop | `NEXTAUTH_URL` mismatch | Ensure `NEXTAUTH_URL` in frontend runtime matches frontend URL (`http://localhost:3000`) and backend `NEXTAUTH_URL` matches backend URL |
| Container exits immediately | Entrypoint script exits after nginx because `exec` not used correctly | Ensure `exec node server.js` is last line; nginx runs in background |
